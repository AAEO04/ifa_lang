Type System

ifa-types - Shared Foundation for All Runtimes

Overview

The ifa-types crate provides the canonical shared type system used across all Ifá-Lang runtime components. It ensures type consistency between the interpreter, bytecode VM, transpiler, and standard library.

"This is the ONE error type. ifa-core, ifa-std, ifa-cli all re-export it. If you need a different error shape, use a different type name - never another IfaError."
Single source of truth principle enforced across all crates

Core Architecture

Module Structure

ifa-types Modules 
ifa-types/
    src/
        lib.rs           // Main exports and crate configuration
        domain.rs        // OduDomain enum (16 Odù + infrastructure)
        error.rs         // IfaError / IfaResult (canonical error types)

        token.rs         // ResourceToken (resource management)

        value_union.rs   // IfaValue (new tagged union - current)

        bytecode.rs      // Bytecode structures (vm feature)

Feature Flags

  • vm: Include AST and bytecode modules
  • std: Enable standard library extensions
  • alloc: Enable heap allocation support
  • default: Core types only (no_std compatible)

IfaValue - Dynamic Type System

The universal container for Ifá-Lang's dynamic typing. The implementation has evolved from an enum to a tagged union for better performance.

Current Implementation (Tagged Union)

The value_union.rs module provides the new tagged union implementation:

IfaValue Types 
// Current IfaValue (from value_union.rs)
pub enum IfaValue {
    // Primitive types
    Int(i64),
    Float(f64),
    Str(Arc),
    Bool(bool),
    
    // Collection types
    List(Vec),
    Map(HashMap, IfaValue>),
    
    // Object types
    Object(Rc, IfaValue>>>), // "The Hut" - thread-local
    
    // Function types
    Fn(IfaFn),        // Standard/Native functions
    AstFn(Statement), // AST functions (vm feature)
    BytecodeFn(Bytecode), // Bytecode functions (vm feature)
    
    // Resource types
    Resource(ResourceToken), // Opaque handle
    
    // Special types
    Ptr(*mut u8),     // Raw pointers (unsafe)
    Ref(*const u8),   // Immutable references
    
    // Control flow
    Return(Box), // Return values
    Null,             // Null value
}

Performance Optimizations

  • Arc<str>: String interning and sharing
  • "The Hut": Thread-local objects with zero atomic overhead
  • Tagged Union: Efficient memory layout and dispatch
  • Resource Tokens: Opaque handles for safe resource management

Legacy Implementation

The value.rs module contains the older enum implementation, now deprecated but kept for compatibility:

  • Basic Types: Int, Float, Str, Bool
  • Collections: List, Map, Object
  • Functions: Fn only (no AST/Bytecode variants)
  • Simpler: Fewer variants, less complexity

IfaError - Canonical Error System

The single authoritative error type used across all Ifá-Lang components. Embodies the Okanran (Errors) domain with Yoruba proverbs.

Error Categories

Math Errors (Obara / Oturupon) - ErrorCode 0x03XX

  • DivisionByZero: "Obara rejects: {context}"
  • Overflow: Numeric overflow conditions
  • Underflow: Numeric underflow conditions

Type Errors - ErrorCode 0x02XX

  • ArityMismatch: Wrong number of arguments
  • ArgumentError: Invalid argument values
  • TypeMismatch: Expected vs actual type conflict
  • ConversionError: Type conversion failures

Index/Key Errors (Ogunda / Ika) - ErrorCode 0x01XX

  • IndexOutOfBounds: Array/collection index errors
  • KeyNotFound: Map key missing errors

Runtime Errors - ErrorCode 0x04XX

  • RuntimeError: General runtime failures
  • MemoryError: Memory allocation failures
  • StackOverflow: Call stack exceeded

Error Integration

Error Usage 
// Result type alias
pub type IfaResult = Result;

// Usage across all crates
use ifa_types::{IfaError, IfaResult};

fn divide(a: i64, b: i64) -> IfaResult {
    if b == 0 {
        Err(IfaError::DivisionByZero("Cannot divide by zero".to_string()))
    } else {
        Ok(a / b)
    }
}

Serialization Support

serde Integration

Most types support serde serialization/deserialization:

  • OduDomain: Enum serialization
  • IfaError: Error serialization
  • IfaValue: Value serialization (with limitations)

Serialization Limitations

  • Functions: Cannot serialize function pointers
  • Objects: Thread-local objects skipped
  • Resources: Opaque handles not serializable
  • Pointers: Raw pointers not serializable

Usage Guidelines

Import Patterns

Recommended Imports 
// Standard imports for most crates
use ifa_types::{
    IfaValue, IfaError, IfaResult,
    OduDomain, IfaShared, ResourceToken
};

// VM-specific imports
#[cfg(feature = "vm")]
use ifa_types::{Statement, Bytecode};

// All traits (convenient)
use ifa_types::traits::*;

Best Practices

  • Single Source: Always import from ifa-types, never redefine
  • Error Handling: Use IfaResult for all fallible operations
  • Domain Types: Use OduDomain for domain-specific operations
  • Resource Safety: Use ResourceToken for managed resources

Performance Considerations

  • Arc<str>: Prefer string internment for repeated values
  • "The Hut": Use Object for thread-local data
  • Tagged Union: More efficient than enum dispatch
  • Feature Flags: Disable VM features when not needed

Migration and Compatibility

Legacy Support

  • value.rs: Old enum implementation kept for compatibility
  • value_union.rs: New tagged union (preferred)
  • Re-exports: lib.rs exports the new implementation

Breaking Changes

  • IfaValue: Switch from enum to tagged union
  • Function Types: Added AST and bytecode variants
  • Resource Management: Enhanced ResourceToken system

Future Directions

  • 256 Odù Types: Support for long-term type lattice
  • Protocol Traits: Enhanced protocol system
  • Performance: Further optimization of tagged union
  • Serialization: Enhanced serde support

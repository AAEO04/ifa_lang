Ifá-Lang Compared

    Type Safety, Philosophy, and Performance

    vs. TypeScript

    Feature TypeScript Ifá-Lang
    Type System Static (Compile-time) Dynamic (Runtime)
    Safety Checks types before running. Type erasure at runtime. Runtime Type Checking. Values carry type info.
    Philosophy "JavaScript with Types" Domain-Driven Standard Library (16 Odù)
    Contracts None built-in (needs libraries) Built-in ewo (assert) for invariants.
    Linter/Static Analysis ESLint (External) Babalawo (Built-in). Checks unused vars, logic errors, and architectural violations.

    Verdict: Use TypeScript for compile-time safety in large codebases. Use Ifá-Lang for a Python-like development experience with the performance and safety of a Rust/WASM runtime.

    vs. Python

    Ifá-Lang feels very similar to Python but compiles to optimized bytecode. The key difference is the Runtime Model:

    Feature Python Ifá-Lang
    Concurrency GIL (Global Interpreter Lock). Threading is limited. Multithreading via Osa (Rayon). No GIL.
    Sandboxing No native sandbox. Requires containers (Docker). Wasm Sandbox via ifa-sandbox (Wasmtime/WASI).
    Mobile/Embedded MicroPython (separate subset) no_std Runtime via `ifa-embedded`. Targets ESP32/STM32.

    vs. JavaScript / Node.js

    Feature Node.js Ifá-Lang
    Web Native (V8 Engine) WASM via `ifa-wasm`. Runs in browser alongside JS.
    Package Manager npm (Huge ecosystem, supply chain risks) Ọjà (Market). Curated, decentralized registry built in.

    vs. Rust

    Ifá-Lang is written in Rust and exposes its power through high-level domains:

    • Macros: Rust has `macro_rules!`. Ifá-Lang has ifa-macros for cultural enforcement (like `iwa_pele`).
    • Ownership: Rust has strict borrow checker. Ifá-Lang has Ẹbọ (Sacrifice) - a simplified resource scope system.

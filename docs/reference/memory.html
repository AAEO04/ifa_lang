Memory Management

Opon System - Sacred Memory Management

Overview

Ifá-Lang uses a sophisticated memory management system inspired by Yoruba divination practices. The system consists of three main components: Opon (memory allocation),  (resource lifecycle), and Àjõsé (reactive relationships).

Opon Memory System

Opon (the divination tray) is Ifá-Lang's primary memory management system, implemented in ifa-core/src/opon.rs (16,003 bytes).

Sacred Memory Philosophy

  • Sacred Space: Memory is treated as sacred space for computation
  • Ritual Cleanup: Resources must be properly returned ( )
  • Pattern Recognition: Memory allocation follows sacred patterns
  • Community Sharing: Memory is shared but respected

Memory Allocation Strategies

Opon Memory Configuration 
// Memory size directives
opon: kekere;              // Small memory - embedded systems
#opon arinrin;               // Medium memory - standard applications  
mem: nla;                   // Large memory - data processing
#mem ailopin;               // Unlimited memory - development

// Runtime memory management
let opon = Opon::new(MemorySize::Medium);
opon.allocate(size);        // Sacred allocation
opon.deallocate(ptr);      // Ritual cleanup

Heap Management

  • Garbage Collection: Automatic cleanup of unreferenced objects
  • Memory Pools: Pre-allocated pools for common sizes
  • Fragmentation Handling: Compaction and defragmentation
  • Leak Detection: Runtime leak detection and reporting

 (Ebo) Resource System

 (sacrifice/offering) manages resource lifecycle with obligation tracking, implemented in ifa-core/src/ebo.rs (5,473 bytes).

Resource Obligation

Every resource acquired through an Odù domain carries an  (obligation) - a sacred duty to properly release the resource:

Resource Sacrifice 
// Resource acquisition with obligation
ebo file = Odi.si("data.txt");

// Use resource
Ika.process(file);

// Automatic sacrifice when scope ends
// File is automatically closed

Resource Types

  • File Handles: Odi domain file resources
  • Network Connections: Otura domain socket resources
  • GPU Resources: Ose domain graphics resources
  • Crypto Contexts: Irete domain crypto resources

Automatic Cleanup

Ebo Block Pattern 
ebo_block! {
    let conn = Otura.so("example.com", 80);
    let file = Odi.si("data.txt");
    
    // Work with resources
    process_data(&conn, &file);
    
    // All resources auto-cleanup when block exits
    // Connection closed, file closed
}

Àjõsé Reactive System

Àjõsé (gathering/assembly) manages reactive relationships and dependency tracking, implemented in ifa-core/src/ajose.rs (19,644 bytes).

Reactive Relationships

Values can be bound together in reactive relationships where changes propagate automatically:

Reactive Binding 
// Create reactive binding
ajose!(counter.value => label.text);

// When counter changes, label updates automatically
counter.value = 42;  // label.text becomes "42"

Dependency Tracking

  • Change Detection: Automatic detection of value changes
  • Propagation: Efficient change propagation through relationships
  • Cycle Detection: Detection and handling of circular dependencies
  • Batch Updates: Batched updates for performance

Memory Efficiency

  • Weak References: Prevent memory leaks in reactive graphs
  • Subscription Cleanup: Automatic cleanup of unused subscriptions
  • Change Batching: Reduce memory pressure from frequent updates

Type-Specific Memory Management

IfaValue Memory Layout

Based on the tagged union implementation in ifa-types/src/value_union.rs:

Value Memory Structure 
// Tagged union layout (efficient)
pub enum IfaValue {
    Int(i64),                    // 8 bytes
    Float(f64),                  // 8 bytes
    Str(Arc),              // 8 bytes + shared string data
    Bool(bool),                  // 1 byte
    List(Vec),        // 24 bytes + elements
    Map(HashMap, IfaValue>), // 24 bytes + entries
    Object(Rc, IfaValue>>>), // "The Hut"
    // ... other variants
}

"The Hut" Pattern

Thread-local objects with zero atomic overhead:

  • Thread-Local: No atomic operations needed
  • Mutable: Interior mutability through RefCell
  • Efficient: Zero-cost access for thread-local data
  • Sacred Space: Protected memory area for thread-local state

String Interning

  • Arc<str>: Shared string references
  • Interning: Common strings are interned
  • Memory Savings: Reduced duplication of string data
  • Fast Comparison: Pointer comparison for interned strings

Memory Safety Features

Ìwà Pélé Balance Checking

Compile-time resource balance checking from ifa-core/src/iwa_pele.rs (7,992 bytes):

Balance Checking 
#[iwa_pele]
fn network_task() {
    let conn = Otura.so("example.com", 80);  // open
    // ... work with connection
    conn.pa();  // close - REQUIRED or compile error
}

// This causes compile error:
#[iwa_pele]
fn bad_function() {
    let file = Odi.si("data.txt");  // open file
    // Missing: file.ti() - compile error!
    // Ìwà Pélé violation: Block leaks resource
}

Best Practices

Resource Management

  • Use Ebo Blocks: Scope resources with ebo_block!
  • Balance Resources: Ensure paired operations (so/pa, si/ti)
  • Early Release: Release resources as soon as possible
  • Avoid Leaks: Use Ìwà Pélé checking to prevent leaks

Memory Efficiency

  • Prefer Stack: Use stack allocation for small objects
  • String Interning: Use interned strings for repeated values
  • Avoid Cloning: Use references when possible
  • Batch Operations: Batch memory operations for efficiency

Reactive Programming

  • Weak References: Use weak references to prevent cycles
  • Subscription Cleanup: Clean up unused subscriptions
  • Batch Updates: Batch reactive updates
  • Avoid Excessive Bindings: Limit reactive binding complexity
